# Go Rewrite — Lessons Learned

Last updated: 2026-07-19

---

## Phase 1 — Foundation (config, server, health, db, cache, telemetry)

### Do not pin indirect dependencies with pseudo-versions in go.mod

When bootstrapping the Go module, `go get` was used to pull specific packages. Some transitive dependencies were pinned with hand-edited pseudo-versions (e.g., `v0.0.0-20230101...`). This caused `go mod tidy` to reject the module and broke `go build`. The correct approach is to only list direct imports in the `require` block of `go.mod` and let `go mod tidy` resolve transitive dependencies automatically. Never manually add a `// indirect` dependency with a pseudo-version unless it was generated by `go mod tidy` itself.

### Test infrastructure before implementing features

The Phase 1 health handler and server tests served as proof that the build toolchain, module resolution, and test runner all worked before any feature code was written. This paid off in Phase 2 — we had a stable baseline to build on. Start every phase with a "does `go test ./...` pass before I touch anything" check.

### Config is a zero-dependency leaf — keep it that way

`internal/config` has no imports from the rest of `internal/`. Adding auth key parsing to config required importing `crypto/rsa`, `crypto/x509`, and `encoding/pem` — all stdlib, so no new module dependency was introduced. The rule to enforce is: config may import stdlib only. Any non-stdlib import in config creates a risk of circular imports and makes startup validation depend on external packages.

### Server signature changes break callers immediately

Changing `server.New` and `server.NewRouter` to accept an `AuthMiddlewares` parameter required updating both `main.go` and `server_test.go` in the same commit. The compiler catches this, which is good — but it means server signature changes must always be accompanied by a search for all call sites (`grep -r "server.New"`, `grep -r "server.NewRouter"`). Go's static typing makes this safe but the refactor still has a blast radius.

## Phase 2 — Local JWT + Bearer Token Cache with Redis Pub/Sub Invalidation

### Use stdlib crypto only for JWT — no third-party library needed

RS256 JWT validation using only `crypto/rsa`, `crypto/x509`, `encoding/pem`, `crypto/sha256`, and `encoding/base64` requires about 60 lines of clear, auditable code. A third-party library would add a module dependency, hide the algorithm selection logic, and introduce a supply-chain risk for a security-critical path. The implementation: split on `.`, base64url-decode each segment, parse header to confirm `alg=RS256`, parse payload JSON to claims struct, SHA256-hash `header.payload`, call `rsa.VerifyPKCS1v15`. That is the complete algorithm.

### Keep auth injectable via interfaces — never import live infra in unit tests

`TokenQuerier` and `RedisClient` are interfaces in `internal/auth`. The production implementations (`*pgxpool.Pool` and `rueidis.Client`) are wired in `main.go`. Tests supply `mockTokenQuerier` and `mockRedis`. This pattern means the entire auth package can be tested with zero live infrastructure, and the mocks exercise the exact same interface contract as production. Violating this rule — even once, by adding a `sync.Map` side-channel that tests reach into — makes tests brittle.

### Two-level cache: L1 eviction via pub/sub only invalidates L1, not L2

The cross-pod revocation flow works as follows:
1. The revoking pod calls `Revoke(ctx, rawToken)`.
2. `Revoke` deletes the L2 Redis key (`them:session:token:{hash}`).
3. `Revoke` publishes `{hash}` on `them:token:revoked`.
4. ALL pods (including the revoker) receive the pub/sub message and evict from L1.

A test that simulates step 4 without first simulating steps 2-3 will not trigger a DB fallback — the second `Validate` call will hit L2 and return the (now-stale) cached result. Tests must simulate the full revocation flow (delete L2 first, then publish) to correctly exercise the cross-pod path.

### pub/sub handler registration race in tests: poll, don't sleep

The `cache.Subscribe(ctx)` goroutine appends its handler to the Redis subscriber list inside `rdb.Subscribe`. Tests that start the goroutine and immediately call `rdb.Publish` will race — the goroutine may not have appended its handler yet. The correct pattern is to poll a `subscriberCount()` accessor on the mock until it returns > 0, with a reasonable deadline (2 s). A fixed `time.Sleep` is fragile under CI load; polling is deterministic.

### Fail fast on malformed JWT_PUBLIC_KEY_PEM at startup

Config validation parses `JWT_PUBLIC_KEY_PEM` immediately. If the PEM is malformed or encodes a non-RSA key, `Load()` returns an error and the process never starts. This prevents a deployment where the pod starts, serves unauthenticated traffic for hours, and only fails on the first JWT validation attempt. Fail fast at startup or not at all.

### Health and metrics routes must remain unauthenticated — use a sub-router

When wiring auth middleware into the chi router, the pattern is a `r.Group(func(protected chi.Router) { ... })` sub-router that applies JWT and Bearer middleware only to protected paths. The top-level router mounts `/health/*` and `/metrics` before any auth middleware so they are always reachable without a token. Liveness probes that fail because the JWT key expired are a production incident waiting to happen.

### JWT claims field names must match the issuer

The Python auth_service issues JWTs with `user_name` (not `username`). The Go `Claims` struct uses `json:"user_name"` to match. Mismatched field names silently produce zero values (empty string / 0) instead of errors, making them hard to detect. Always cross-reference the issuer's source before naming JSON fields in the consumer.

## Phase 3 — Session Manager

### Atomic Lua scripts are the only safe way to combine multiple Redis operations

The session gate must atomically check the session count, increment it, and create a shadow TTL key in one operation. Any gap between these steps (e.g., two separate `SET` + `EXPIRE` calls) creates a window where a concurrent pod can observe inconsistent state. Lua scripts run atomically on the Redis server — no other commands can interleave. Use a Lua script for any "read-modify-write" or "check-and-set" pattern on Redis.

### Shadow TTL keys are the ghost-detection mechanism

The session set (`them:session:ep:{id}:members`) is not TTL-based — it is a plain Redis Set. Each member key (`them:session:ep:{id}:member:{session_id}`) is a separate TTL key. When a pod dies without cleanup, the member set retains the orphaned `session_id` but the member key expires. Ghost pruning checks each member for the presence of its key; missing = ghost; remove from set and decrement counter. Always prune on every gate call, not only when the cap is full — a missed ghost grows the effective session count above the cap.

### Pod heartbeat must report real active sessions, not a hardcoded constant

The original Python bridge hardcoded `active_sessions=0` in heartbeat payloads. The Go rewrite uses `atomic.LoadInt32(&activeSessions)` to report the real count. Correct counts are required for the dashboard heat map and for capacity planning. The atomic counter is incremented on `Gate` (claim) and decremented on `Release` (cleanup) — never in any other path.

## Phase 4 — Event Bus, Domain Types, Run Recorder

### In-process fan-out bus: hold a read lock during Publish for concurrent safety

The event bus map (`topic → []subscriber`) must be read-locked during `Publish` so that multiple publishers can proceed concurrently. A write lock would serialise all publishers, which is unnecessary because `Publish` only reads the map (it does not add or remove subscribers). Write-lock only in `Subscribe` and the unsubscribe function. This is the standard "many readers, few writers" pattern.

### Drop on full channel, never block — bus must never block on a slow consumer

The bus uses `select { case s.ch <- e: default: }` for every subscriber send. If the subscriber's buffer is full the event is silently dropped for that subscriber. Blocking would propagate back-pressure from a single slow WebSocket client to all publishers, including Temporal activity callbacks. The trade-off is intentional: real-time streaming events (token deltas) are not worth queuing indefinitely.

### Buffered mock channel of size 1 — not len(responses) — for cancellation tests

When writing the MockProvider (Phase 5), using `make(chan StreamEvent, len(responses))` as the internal buffer caused all events to be enqueued before the goroutine's `select` could observe context cancellation. Using a buffer of 1 forces the goroutine to block on each send, making the `ctx.Done()` arm reachable after each event. This is the correct pattern for testing "does cancellation stop a stream mid-way?"

### DBQuerier interface isolates recorder from pgxpool — no live DB in tests

`runrecorder.DBQuerier` exposes only `Exec(ctx, sql, args...) error`. The mock captures calls in a slice. Tests assert SQL fragments and argument ordering without starting Postgres. The interface is intentionally narrow — only the methods the recorder actually uses. Wider interfaces (e.g., including `Query`) create unused surface that mocks must implement.

### Wrap DB errors with context (runID, step type) before returning

All recorder methods return `fmt.Errorf("runrecorder: CreateRun %s: %w", run.ID, err)`. This means the caller's error log always includes the run ID and operation name without needing to repeat them at every call site. The `%w` verb preserves `errors.Is` unwrapping so callers can check for sentinel errors if needed.

## Phase 5 — LLM Provider Abstraction

### Provider interface is the only LLM abstraction — no SDK, no shared globals

`llm.Provider` takes `[]domain.Message` and `[]llm.ToolDef` (canonical types), not Anthropic-specific structs. Provider implementations translate at the boundary. This is the fix for High finding #7 (provider format leaked into DB) and ensures that swapping providers requires changing only the wiring in `main.go`, not orchestration logic.

### ctx cancellation on HTTP streaming: use NewRequestWithContext, not req.WithContext

`http.NewRequestWithContext(ctx, ...)` binds the context at construction. When ctx is cancelled, the underlying TCP connection is closed by the http transport, which causes the SSE `bufio.Scanner` reading the response body to return an error on the next `Scan()` call. The goroutine then exits, the channel is closed, and the caller's `range ch` loop terminates. No extra plumbing is needed — the stdlib handles it.

### SSE parsing: "data: " prefix, blank line separators, "[DONE]" terminator

Anthropic's streaming API uses standard SSE framing. Each event is a line beginning with `data: ` followed by a JSON object. Events are separated by blank lines. The stream ends with `data: [DONE]`. When `bufio.Scanner` hits a `message_stop` event type in the JSON payload, close the channel immediately rather than waiting for EOF — the stream may keep the connection open briefly after the logical end.

### ToolDef.Validate() at the call site, not inside Stream()

`Stream()` does not validate tools. The caller is responsible for calling `td.Validate()` before passing tools to `Stream()`. This keeps the hot path (streaming) free of error-return allocations and makes the validation boundary explicit. In orchestration code, validate tools once at startup/config-load, not on every LLM call.
